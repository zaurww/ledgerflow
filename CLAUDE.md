## Проект: LedgerFlow

**Что это:** Open-source бухгалтерская система для малого и среднего бизнеса.
**Основатель:** один человек, не программист, с глубоким знанием бухгалтерии.
**Цель:** заменить 1С и закрытые системы для компаний в СНГ/Азербайджане.
**Лицензия:** AGPL-3.0
**GitHub:** https://github.com/zaurww/ledgerflow

## Принципы которые НИКОГДА не нарушаются

1. **Простота важнее умности** — если есть два решения, выбираем более простое
2. **Читаемость важнее краткости** — код читается людьми, не только машинами
3. **Модульность** — каждый модуль независим, общается через события
4. **Факты неизменны** — записанное событие никогда не удаляется, только сторнируется
5. **Сначала работает, потом оптимизируется** — не оптимизируем то чего нет
6. **Один человек** — решения принимаются с учётом что поддерживает один человек

## Архитектурные решения (зафиксированы, не пересматривать без причины)

### Dimensions (аналитические измерения)

- Dimensions хранятся как JSONB колонка в journal_entry_lines
- Используются для произвольных тегов: номер с/ф, департамент, и т.д.
- Правило: если у сущности есть свои поля (бюджет, менеджер) — отдельная таблица
- Правило: если это просто тег для фильтрации — JSONB dimensions

### Иерархия документов (реализуется в Фазе 3)

Структура: projects → contracts → invoices

- invoice_id на строке проводки — FK в таблицу invoices
- Через invoice автоматически известен contract и project
- Не дублировать project_id и contract_id отдельно на строке проводки

### Контрагенты

- Отдельная таблица counterparties — это полноценная сущность, не тег
- counterparty_id — отдельное поле на строке проводки (не в JSONB dimensions)
- В CoA контрагент не фигурирует — это аналитика на проводке, не счёт

### Локализация данных

- В таблицах данных: две колонки name_az (основной язык) и name_en
- Русский язык не закладывается отдельным полем сейчас — добавить миграцией позже если нужно
- Язык интерфейса (i18n) — решается на фронтенде через i18next, не в этой фазе

### Главные архитектурные правила

Правило 1: Направление зависимостей

web → api → modules → core
(только вниз, никогда вверх)

core не знает про modules. modules не знают друг про друга напрямую.
Общение между модулями — только через Event Bus в core/events/.

Правило 2: Контракт модуля
Каждый модуль обязан иметь файл module.py с классом:

class SomeModule:
    def register(self) -> ModuleInfo:      # имя, версия, события
    def get_routes(self) -> list[Router]:  # API роуты
    def get_migrations(self) -> list:      # миграции БД

Правило 3: События, не прямые вызовы

# НЕЛЬЗЯ (модули знают друг о друге):
from modules.inventory import update_stock
update_stock(sku, qty)

# МОЖНО (через событие):
await event_bus.emit("inventory.goods_received", {sku: sku, qty: qty})

Правило 4: Факты неизменны

# НЕЛЬЗЯ (удалять или редактировать проводки):
db.delete(journal_entry)
journal_entry.amount = new_amount

# МОЖНО (только сторнирование):
await ledger.reverse(journal_entry_id, reason="correction")

## Соглашения по коду

Именование:
- Файлы: snake_case (journal_entry.py)
- Классы: PascalCase (class JournalEntry)
- Функции и переменные: snake_case (def create_journal_entry)
- Константы: UPPER_SNAKE_CASE (MAX_BATCH_SIZE = 5000)

Структура файла (порядок всегда такой):
1. Импорты стандартной библиотеки
2. Импорты сторонних библиотек
3. Импорты из core (если модуль)
4. Константы модуля
5. Модели данных (Pydantic)
6. Модели базы данных (SQLAlchemy)
7. Сервисный слой (бизнес-логика)
8. Вспомогательные функции

Комментарии объясняют ПОЧЕМУ, не ЧТО:
# ПЛОХО: account.balance += amount  # прибавляем сумму
# ХОРОШО: account.balance += amount  # дебет увеличивает активные счета (тип Asset)

## Бухгалтерские правила в коде

Двойная запись:
- Каждая транзакция: сумма дебетов = сумма кредитов
- Проверка происходит ВСЕГДА перед записью в GL

Типы счетов и нормальное сальдо:
- Assets (Активы):       нормальное сальдо — ДЕБЕТ
- Liabilities (Пассивы): нормальное сальдо — КРЕДИТ
- Equity (Капитал):      нормальное сальдо — КРЕДИТ
- Revenue (Доходы):      нормальное сальдо — КРЕДИТ
- Expenses (Расходы):    нормальное сальдо — ДЕБЕТ

Денежные суммы — ВСЕГДА Decimal, НИКОГДА float:
- Правильно: amount = Decimal("1000.50")
- НЕЛЬЗЯ:    amount = 1000.50  (ошибки округления)

## Как работать с этим проектом (инструкция для Claude)

### Начало каждой сессии

1. Прочитать этот файл (CLAUDE.md)
2. Прочитать CHANGELOG.md — что было сделано последним
3. Спросить пользователя что делаем сегодня
4. Проверить что задача вписывается в текущую фазу

### При добавлении нового кода

1. Определить: это core или module?
2. Создать тест ПЕРЕД кодом (или одновременно)
3. Следовать структуре файла (см. выше)
4. Добавить запись в CHANGELOG.md
5. Обновить статус в этом файле (раздел "Что реализовано")

### При добавлении нового модуля

1. Создать папку modules/имя_модуля/
2. Создать module.py с классом модуля (контракт)
3. Создать models.py — модели БД
4. Создать schemas.py — Pydantic схемы
5. Создать service.py — бизнес-логика
6. Создать router.py — API эндпоинты
7. Создать tests/modules/test_имя_модуля.py
8. Зарегистрировать в api/main.py

### Что НИКОГДА не делать

* Не писать бизнес-логику в роутерах (только в service.py)
* Не обращаться к БД из роутеров напрямую
* Не использовать float для денег
* Не удалять проводки (только сторнировать)
* Не делать модули зависимыми друг от друга напрямую
* Не усложнять то что работает просто

## Часто задаваемые вопросы

Q: Нужно добавить новый тип операции. С чего начать?
A: Определить событие → написать правило проводки в core/rules/ → создать эндпоинт в нужном модуле → написать тест.

Q: Пользователь хочет фичу которая нарушает архитектуру. Что делать?
A: Объяснить почему это нарушает архитектуру и предложить альтернативу которая вписывается.

Q: Что делать если нужно изменить существующую проводку?
A: Создать сторнирующую проводку (reversal) и новую правильную. Оригинал не трогать.

Q: Модуль нужен "прямо сейчас" но он не в текущей фазе?
A: Создать заглушку (stub) — пустой модуль с правильной структурой. Реализацию — в следующей фазе.

Q: Контрагент — это счёт в CoA?
A: Нет. Контрагент — отдельная таблица, counterparty_id поле на строке проводки. В CoA не фигурирует.

Q: Когда использовать JSONB dimensions, когда отдельную таблицу?
A: Если сущность имеет свои поля (бюджет, статус, менеджер) — отдельная таблица. Если просто тег для фильтрации — JSONB dimensions.
